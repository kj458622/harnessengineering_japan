Codex를 활용한 하네스 엔지니어링으로 서비스 개발은 어떻게 진행될까? AI 인터뷰를 통해 요구사항을 구체화하고, 계획 수립, 목업 UI 구현, API 연동, 배포까지 전 과정을 거치며 **AI 기반 개발 워크플로우**를 구축할 수 있습니다.

## 1. Codex 하네스 엔지니어링 개요

Codex 하네스 엔지니어링은 AI 에이전트의 작동 방식을 고정하는 구조를 만들어 원하는 대로 동작하게 하는 기술이다.

## 1.1. 하네스의 개념 및 중요성

!captureSource

1. **하네스**는 AI 에이전트가 일하는 방식을 고정하는 구조를 의미한다.
2. 기존의 프롬프트만으로는 복잡한 앱 개발에 한계가 있어 하네스가 필요하다.
3. 하네스를 통해 AI 에이전트가 원하는 방식으로 동작하도록 만들 수 있다.
4. 하네스는 반복성을 만들고, 강제성과 자율성을 분리하며, 팀 단위로 축적될 수 있다는 장점이 있다.
5. 따라서 하네스 엔지니어링은 AI 기반 개발 워크플로우 구축에 매우 중요하다.

## 1.2. Codex 하네스 구성 요소

!captureSource

1. **에이전트.md (Agent.md)**:
    - Codex의 규칙 파일로, 에이전트가 항상 참고할 수 있는 규칙을 정의한다.
2. **메모리 (Memory)**:
    - 기본적으로 비활성화되어 있으나, 활성화 시 별도의 메모리 파일이 생성된다.
    - 코덱스와의 대화 내용 중 배울 점이나 선호 사항을 저장하여 다음 세션에서 참고할 수 있게 한다.
3. **스킬 (Skill)**:
    - 반복적인 절차나 전문성을 패키징하는 기능으로, 프롬프트 템플릿보다 확장된 개념이다.
    - 프롬프트뿐만 아니라 스크립트, 템플릿 등을 포함하여 전문화된 영역을 가질 수 있다.
4. **MCP (Model Context Protocol)**:
    - LM 모델에게 외부 도구를 연결할 수 있게 해주는 표준 프로토콜이다.
    - 엔트로픽에서 나온 개념이지만 Codex에서도 활용 가능하다.
5. **훅 (Hook)**:
    - 규칙 파일, 메모리, 스킬 등을 에이전트가 100% 지키도록 보장하기 어려운 부분을 프로그래밍 로직으로 강제하는 기능이다.
6. **서브 에이전트 (Sub-agent)**:
    - 전문 역할을 가진 별도 에이전트를 생성하여 완전히 별도의 컨텍스트에서 작업하게 한다.

## 1.3. Codex 설정 경로

!captureSource

1. **홈 경로의 .codex**:

2. **홈 경로의 .agent**:

## 2. Codex 하네스 구성 요소 상세 설명 및 실습

Codex를 활용하여 '커밋 히어로'라는 웹 서비스를 개발하는 과정을 통해 하네스 구성 요소들의 실제 적용 방법을 알아본다.

## 2.1. 에이전트.md (Agent.md) 설정 및 활용

!captureSource

1. **전역 및 프로젝트 스코프**:
    - Codex는 전역 규칙과 프로젝트별 규칙을 모두 따를 수 있다.
    - 전역 규칙은 홈 경로의 `.codex/agent.md`에, 프로젝트별 규칙은 각 프로젝트의 `agent.md`에 정의한다.
2. **중첩 구조 및 마스터 프롬프트**:
    - `agent.md`는 중첩 구조로 생성이 가능하며, 루트 경로에 설정하면 모든 상황에서 규칙을 읽는다.
    - 예시로 공유된 마스터 프롬프트는 실습 프로젝트 빌드 시 활용된다.

## 2.2. 메모리 (Memory) 활성화 및 활용

!captureSource

1. **메모리 기능**:
    - 이전 대화와 작업에서 반복적으로 사용할 맥락을 저장하여 이후 세션에서 활용한다.
2. **활성화 방법**:
    - Codex 설정 파일에서 메모리를 활성화하고 관련 설정을 켜면 사용 가능하다.
    - 설정 파일(`config.yaml`)을 직접 수정하거나, Codex 에이전트에게 활성화를 요청할 수 있다.
3. **메모리 저장 내용**:
    - 작업 중 선호 사항, 업무 흐름, 기술 스택, 자주 빠지는 실수 등을 코덱스가 추출하여 저장한다.
4. **'Use Memory' 옵션**:
    - 이 옵션이 켜져 있으면 유용한 컨텍스트가 메모리에서 자동으로 불러와진다.
    - 아직 공식 활성화 기능은 아니므로 별도 활성화가 필요하다.

## 2.3. 스킬 (Skill) 활용

!captureSource

1. **스킬의 역할**:
    - 반복되는 절차를 패키징하는 기술로, 특정 작업에서 코덱스가 안정적으로 따를 수 있는 지시, 자료, 스크립트를 묶은 재사용 가능한 워크플로우이다.
2. **점진적 공개 방식**:
    - 스킬은 한 번에 모든 컨텍스트를 불러오는 방식이 아니라 점진적으로 공개된다.
3. **스킬 구조**:
    - 스킬 폴더 안에 `skill.md`와 필요한 파일들이 포함되며, `skill.md` 내의 프론트 메터에서 컨텍스트를 점진적으로 로딩한다.
4. **모듈화**:
    - 반복적으로 작업해야 하는 워크플로우는 스킬로 분리하여 모듈화하는 것이 좋다.

## 2.4. MCP (Model Context Protocol) 활용

!captureSource

1. **외부 시스템 연결**:
    - 에이전트가 직접 할 수 없는 외부 도구(Slack, Gmail 등)를 연결하는 데 사용된다.
2. **기능 예시**:
    - 메일 읽고 답장하기, 슬랙 메시지 읽고 분석 및 전송하기 등이 가능하다.
3. **실습 적용**:
    - 실습 과정에서 MCP를 활용할 수 있다면 함께 사용해 볼 예정이다.

## 2.5. 훅 (Hook)을 통한 강제 실행

!captureSource

1. **런타임 자동화 및 가드레일**:
    - 에이전트 루프 실행 중 특정 커맨드를 주입할 수 있는 기능이다.
2. **이벤트 기반 동작**:
    - 세션 시작, 프롬프트 제출, 도구 사용 이전, 턴 종료 등 다양한 이벤트에서 강제적으로 동작한다.
3. **프로그래밍 로직 기반**:
    - 규칙 기반이 아닌 프로그래밍 로직으로 동작하며, 하네스를 최종적으로 강제하는 역할을 한다.

## 2.6. 서브 에이전트 (Sub-agent) 활용

!captureSource

1. **병렬 작업 위임**:
    - 별도의 독립적인 컨텍스트를 가진 에이전트를 생성하여 작업을 위임한다.
2. **병렬 요청 및 검증**:
    - 메인 에이전트가 원하는 만큼 서브 에이전트를 생성하여 병렬로 작업을 요청할 수 있다.
    - 작업 속도 향상이나 별도 컨텍스트에서의 검증에 활용된다.

## 2.7. '커밋 히어로' 서비스 빌드 실습

!captureSource

1. **서비스 개요**:
    - GitHub 사용자 이름을 입력하면 개발자 커밋 정보를 분석하여 RPG 캐릭터 카드 형식으로 보여주는 웹 서비스이다.
2. **개발 워크플로우**:
    - **인터뷰**: AI가 질문하여 요구사항을 구체화한다.
    - **플랜**: 인터뷰 기반으로 제약 조건 등을 구체화하여 계획을 수립한다.
    - **목업 UI 구현**: 기본적인 프로젝트 구조 설정 후 규칙 정의 및 AI 개발 진행을 위한 목업 UI를 구현한다.
    - **계획 검토 (HTML 시각화)**: AI가 생성한 계획을 사람이 검토하기 쉬운 HTML 형식으로 변환한다.
3. **Codex 앱 인터페이스**:
    - 채팅 히스토리, 채팅 쓰레드 목록 확인 및 빌드 진행이 가능하다.
4. **폴더 선택 및 프롬프트 작성**:
    - '커밋 히어로'라는 이름으로 바탕 화면에 폴더를 생성하고 러프한 프롬프트를 작성한다.
5. **'Deep Interview' 스킬 활용**:
    - 작성된 러프 프롬프트를 구체화하기 위해 'Deep Interview' 스킬을 사용한다.
    - 인터뷰를 통해 서비스 목표, 카드 톤, 데이터 입력 방식, 카드 요소, 공유 방식, 생성 방식, 배포 구조 등을 결정한다.
6. **목업 UI 구현 요청**:
    - 인터뷰 결과와 구현 계획을 바탕으로 백엔드 연동 없이 프론트엔드 목업 UI 구현을 요청한다.
7. **In-app Browser 활용**:
    - Codex 앱 내장 브라우저를 통해 개발 서버 실행 및 렌더링 화면을 확인한다.
    - 서비스 언어를 한국어로 변경하는 요청도 진행한다.
8. **Codex 설정 조정**:
    - GPT 모델은 GPT 5.5를 사용하고, 인텔리전스는 '매우 높음'으로 설정한다.
    - 속도 옵션은 크레딧 소모량과 관련 있으며, 일상 개발 시에는 '보통'으로 설정하는 것이 좋다.

## 2.8. GitHub API 연동 및 계획 문서 작성

!captureSource

1. **백엔드 API 작업 준비**:
    - GitHub 공개 데이터 연동을 위한 백엔드 API 작업을 진행한다.
    - GitHub API는 퍼블릭 API만 사용하며, 토큰 없이 호출 가능한 REST API로 설계한다.
2. **카드 분석 규칙 확정**:
    - 카드 분석 규칙은 단순 기반 설명 가능한 점수화로 확정한다.
3. **계획 문서 작성**:
    - 인터뷰 종료 후 계획 문서를 작성하도록 요청한다.
    - 장기간 작업 시 에이전트가 흔들리지 않도록 별도의 메모리 역할을 하는 계획 문서가 중요하다.
4. **HTML로 계획 검토**:
    - 마크다운 형식의 계획 문서를 사람이 5분 안에 검토할 수 있는 단일 HTML로 변환하여 시각화한다.
    - 이는 휴먼 리뷰 단계에서 유용하다.
5. **계획 문서 저장 위치**:
    - 계획 문서는 레포 경로에 저장하는 것이 프로젝트별 관리에 효과적이다.

## 2.9. API 연동 결과 확인 및 배포

!captureSource

1. **API 연동 결과 확인**:
    - GitHub 사용자 ID를 입력하면 커밋 정보 기반으로 분석 결과가 카드 형식으로 표시된다.
    - URL 공유 및 PDF 다운로드 기능이 제공된다.
2. **새로운 세션 및 Worktree 활용**:
    - API 연동 후 AI 기능 추가를 위해 새로운 세션을 열고 'Worktree'를 활용하여 병렬 작업을 진행한다.
    - AI 기능은 OpenAI GPT 모델을 사용하며, 요청 시 비용이 부과된다.
3. **GitHub Repository 푸시**:
    - 작업 완료 후 메인 브랜치로 푸시하여 GitHub Repository에 코드를 올린다.
4. **Vercel 플러그인을 통한 배포**:
    - Codex의 Vercel 플러그인을 사용하여 GitHub Repository를 Vercel에 배포한다.
    - 플러그인은 MCP와 스킬의 조합으로, 설치 및 사용이 용이하다.
5. **배포 완료 확인**:
    - 배포된 Vercel 앱에 접속하여 서비스가 정상적으로 작동하는지 확인한다.

## 2.10. 환경 변수 관리 및 AI 기능 연동

!captureSource

1. **환경 변수 보안**:
    - 환경 변수 파일은 에이전트가 접근하지 못하도록 '훅'을 사용하여 차단한다.
2. **AI 기능 연동**:
    - AI 기능 연동이 완료되면 GitHub API의 시간당 호출 제한으로 인해 일시적으로 작동하지 않을 수 있다.
3. **전체 빌드 과정 요약**:
    - 인터뷰, 계획 수립, 목업 UI 구현, API 연동, AI 기능 추가, GitHub 푸시, Vercel 배포까지 전체 과정을 진행했다.
4. **핵심 개발 흐름**:
    - 명확한 프롬프트 작성을 위해 AI의 도움을 받아 인터뷰 단계를 거쳐 요구사항을 구체화하고, 이를 기반으로 계획을 세운 후 개발을 진행하는 흐름을 따른다.
5. **Agent.md 설정**:
    - 공유된 마스터 프롬프트를 활용하여 `agent.md`를 설정하고 스킬로 설치하여 활용할 수 있다.
